In RAMADDA when there is a collection of checkboxes in a form (e.g., in the entry list or in the edit metadata form) selecting one and then selecting another with a Shift-Click toggles the range of checkboxes between the first the second checkbox clicked. Doing a Control-Click on a checkbox toggles all of them on or off. To configure SSL access see here if you are running under Tomcat. See here if you are running stand-alone. For security reasons you can configure what IP addresses users are allowed to login from. In your repository.properties file in your local RAMADDA directory, e.g.,
<home>/.unidata/repository/repository.properties
add the property:

##Comma separated list of ip addresses (or prefixes) that are the ones where the user can login
##If blank then login is allowed from anywhere
ramadda.login.allowedips=
##e.g.:
##ramadda.login.allowedips=127.0.0,201.43.72.5
You need to comment out the ramadda.ssl.port property in your .properties file. Alternatively, you can run RAMADDA with the command line argument: 
-Dramadda.ssl.ignore=true
 You can specify an administrator user id and password from the command line when starting RAMADDA. If you are running RAMADDA stand alone then just provide the arguments:
-admin <user id> <password>
This creates a new user id if needed and sets the password.

If you are running RAMADDA under Tomcat then change the web.xml to pass in command line arguments using the "args" entry. This is a comma separated value list of the command line arguments that are processed by RAMADDA. 

    <param-name>args</param-name>
    <param-value>-admin,userid,password</param-value>

You can also set a username/password in the repository.properties file in the RAMADDA home directory. Just add:

ramadda.admin=userid:password
and then restart RAMADDA. This sets the password for the given userid. Note: you should remove this entry once you have reset your password. RAMADDA can send email for both the password reset process and for monitoring new entries. On the main Site Administration page under the "Contact" area you can specify a SMTP server that RAMADDA will use to send email. The HTML pages that RAMADDA generates are based on a template mechanism. Here is an example template: template.html.txt. Copy the example to your local RAMADDA directory (strip off the .txt):
<user home>/.unidata/repository/resources/template.html
This is one of the places where RAMADDA looks for templates.

In the template are a set of properties. You should change the id and name property. There are a number of other macros shown as ${macro} that are used to instantiate a web page. The ${content} is where the generated html is placed.

The ${root} is the root directory path used by RAMADDA, e.g., "repository". You can reference files (e.g., style sheets, images) with this path, e.g.:

    <img src="${root}/someimage.png">
When RAMADDA is serving a file (e.g., someimage.png) it will look in: <user home>/htdocs So place any files you want served in that directory.

RAMADDA uses the "ramadda.html.templates" property, a semi-colon delimited list of file paths, to find where the templates are. The default value for this property is: 

    ramadda.html.templates=%repositorydir%/resources/template.html;%resourcedir%/templates/wikipedia.html;...
If you wanted to change this then make some .properties file with the new paths, e.g.,:
    ramadda.html.templates=%repositorydir%/resources/template1.html;%repositorydir%/resources/template2.html;...
and place the .properties file in the plugins directory:
    <user home>/plugins
Normally, RAMADDA will cache template in memory so if you are modifying the template you can't just hit reload in your browser. You would need to restart RAMADDA. To get around this you can run set the property:
    ramadda.cacheresources=false
in your properties file or specify this property on the command line: 
    java -Xmx1024m -server   ucar.unidata.repository.RepositoryServer -Dramadda.cacheresources=false
Don't forget to unset this property when in production. [ht::foot]
