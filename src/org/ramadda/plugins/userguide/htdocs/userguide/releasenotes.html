1 RAMADDA Version 1.6 Release Notes
RAMADDA version 1.6 is now available for download at:
    http://sourceforge.net/projects/ramadda/files/ramadda1.6/
Note: This release fixes a long-standing but recently discovered issue that results in having user passwords occasionally appear in the server side access logs. See below for more details.

What's new?

  • This release features a new display front-end integrated into the wiki facility - http://ramadda.org/repository/alias/displays. This enables the creation of rich interactive search, charting and data access interfaces across multiple federated repositories.
  • A new Case Study contstruct
  • Extended entry editing including the ability to change entry types.
  • Automatic extraction of thumbnail images from powerpoint and word files.
  • And as usual, lots of bug fixes and additions to the documentation.

Upgrading

Note: Many users ask if upgrading RAMADDA will effect any existing content. All of the content and files are kept in the RAMADDA home directory and in the database. As long as those directories remain untouched there should not be any problem in upgrading. However, you will need to update any plugins you have installed in the RAMADDA plugins directory.

To update an existing RAMADDA install:

  • Update plugins
    • If you have installed the old allplugins.zip then delete the file and install the new allplugins.jar. Plugins are installed in the <ramadda home>/plugins directory. If you currently have allplugins.jar installed then RAMADDA will update it automatically.
    • If you have the pointplugin.jar installed then delete it. It is now provided in the core.
    • If you have other plugins installed then you should update them to the latest versions.
  • Install the new server
    • If you are running stand-alone then download and unzip the ramadda1.6.zip. You will need to adjust your scripts to point to the ramadda1.6 directory.
    • If you are running under Tomcat or Glassfish download and install the repository.war

Password Exposure

Up until and including RAMADDA V1.6B there is the possibility to have clear text passwords written out in the server side access logs. The 1.6 release fixes this problem. We strongly recommend users take the following steps:
  • Upgrade to the latest 1.6 release.
  • Change your RAMADDA passwords.
  • Delete any of the access.log or repository.log files in your ramadda home/logs directory on your server.
  • If you are running RAMADDA from within Apache and/or Tomcat they too may have the passwords exposed in their access logs.

Some RAMADDA html templates may import resources (e.g., style sheets, images) from somewhere other than RAMADDA. In that case, passwords may also be exposed in the REFERER HTTP argument for the resource request. To check grep for "password=" in the logs. Delete the logs if necessary.

There is a new Admin->Maintenance->Clear All Passwords service that allows you to clear all of the passwords held in the RAMADDA user database.

What happened?
When you are visiting any page in RAMADDA the Login link that is shown has an extra redirect argument which is the URL of the page you are viewing. Normally, if you are viewing a page and then go to login, after a successful login you will be redirected back to the original page.

The exposure comes about in the following way:

  • Go to the login page
  • Enter an incorrect password or userid and attempt to login
  • The result page will show the login error. But, the Login link in the upper right will have as its redirect argument the original credentials.
  • In the result page, if you then login correctly through the form there is no problem.
  • However, if you click on the Login link again then login successfully you will be redirected back to the login page with your original credentials showing up in the URL.
My apologies for this problem. I strive to make RAMADDA secure but this slipped through the cracks. If you have any questions please contact me. -Jeff McWhirter
